# Library research

## Considerations for HTTP server library

The HTTP server library needs to support a small REST API, static Swagger UI assets, an OpenAPI specification endpoint, and enough concurrency to avoid blocking the service while longer inventory operations are processed in the background. The project also needs to stay practical to build with CMake, without spending most of the implementation effort on low-level networking infrastructure.

* `cpp-httplib` -- was the first option considered because of its simple usage and small codebase. To integrate it, you only need to include a header, which makes it the simplest option by far. Unfortunately, it is completely blocking and does not use modern event-driven APIs such as epoll, kqueue, or IOCP. This causes handlers to block, which limits the server's concurrent throughput to the number of available worker threads.
* `Oat++` -- was considered as a potential fit, but its development has slowed down due to the war in Ukraine, so it was not picked.
* `Boost.Beast` -- was considered because it is one of the most widely used and technically strong options for C++ HTTP networking. It is built on Boost.Asio and is a good foundation for a production-grade asynchronous HTTP server. The drawback is that Beast is comparatively low-level. Integrating it is not only a build dependency decision; the application would also need to implement or assemble its own routing, request handling structure, static file serving, response helpers, async connection/session lifetime management, error handling, and graceful shutdown behavior. **In a production environment I would likely pick Boost.Beast, but it is considerably more effort than is appropriate for the current project scope.**
* `Drogon` -- was selected as the best practical option for this project. It provides an asynchronous HTTP framework with C++17 as the minimum supported version and C++20 coroutine support when available. It also provides CMake integration, route registration, static file serving, and response callbacks that match the intended architecture better than cpp-httplib. It avoids much of the low-level plumbing required by Boost.Beast while still providing a more scalable event-driven foundation. For this project, Drogon gives the right balance between implementation effort, async capability, and maintainability.

As a result, C++17 remains the minimum acceptable version, but C++20 was chosen for this implementation because it gives access to coroutine support.
